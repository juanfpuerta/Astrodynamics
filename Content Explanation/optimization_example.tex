\section*{Introduction}
When using NASA's General Mission Analysis Tool (GMAT), the \textbf{Yukon} optimizer frequently operates as a ``black box.'' To use it effectively for complex orbital maneuvers, it is crucial to understand its internal mechanics. Yukon is a \textbf{Sequential Quadratic Programming (SQP)} solver. Below is a detailed breakdown of how an SQP algorithm functions, using Python's \texttt{SciPy} library (\texttt{SLSQP}) as a direct analogue to GMAT's Yukon.

% ---------------------------------------------------------
\section*{1. The Objective Function (The ``Valley'')}

This is the direct equivalent of GMAT's \texttt{Minimize} command. We are instructing the algorithm: \textit{``Your primary goal is to make this specific number as small as possible.''}

\begin{lstlisting}[language=Python]
def objective_function(burns):
    burn1, burn2 = burns
    return burn1**2 + burn2**2 
\end{lstlisting}

\textbf{Why square the burns?} In real astrodynamics, we want to minimize absolute $\Delta V$. However, mathematical optimizers struggle with the sharp corners of absolute value graphs (like a ``V'' shape), because the derivative at the point is undefined. By squaring the burn durations ($x^2 + y^2$), we create a smooth, 3D mathematical ``bowl'' (a paraboloid). The optimizer can easily calculate the derivative (the slope) and slide smoothly down to the bottom of the bowl.

% ---------------------------------------------------------
\section*{2. The Non-Linear Constraint (The ``Fence'')}

This section represents GMAT's \texttt{NonLinearConstraint} command, dictating the strict physics boundaries the algorithm must respect.

\begin{lstlisting}[language=Python]
def physics_constraint(burns):
    burn1, burn2 = burns
    achieved_altitude = 20 * burn1 * burn2 + 50 * burn1
    target_altitude = 5000.0
    return achieved_altitude - target_altitude

cons = {'type': 'eq', 'fun': physics_constraint}
\end{lstlisting}

\begin{itemize}
    \item \textbf{The Mock Physics:} The equation inside the function is a non-linear mathematical stand-in for the highly complex orbital mechanics that GMAT calculates when propagating a spacecraft state.
    \item \textbf{The ``Zero'' Rule:} Sequential Quadratic Programming algorithms (like Yukon) require constraints to be formulated as finding a root. We do not tell the algorithm ``equal 5000''. We must subtract the target so the algorithm attempts to drive the output exactly to zero: 
    $$f(x) = \text{Achieved} - \text{Target} = 0$$
    \item \textbf{The Dictionary (\texttt{cons}):} The \texttt{'type': 'eq'} parameter tells the solver this is an \textit{equality} constraint (it must achieve exactly 5,000 km). If it were an \textit{inequality} constraint (e.g., altitude must be \textit{greater} than 5,000 km), we would use \texttt{'ineq'}, granting the optimizer much more freedom.
\end{itemize}

% ---------------------------------------------------------
\section*{3. Initial Guess \& Bounds (The ``Starting Point'')}

This maps directly to GMAT's \texttt{Vary} command logic.

\begin{lstlisting}[language=Python]
initial_guess = [10.0, 10.0]
bnds = ((0, None), (0, None))
\end{lstlisting}

\begin{itemize}
    \item \textbf{The Initial Guess:} Gradient descent algorithms are effectively blind; they only ``feel'' the local slope directly underneath them. If you provide a starting guess of \texttt{[0.0, 0.0]}, the initial slope might be flat (zero). The solver will multiply by zero and immediately fail. You \textit{must} provide the solver with a physically reasonable starting point for it to succeed.
    \item \textbf{The Bounds:} We apply \texttt{(0, None)} to both variables, restricting the burn durations from zero to infinity. This prevents the algorithm from attempting ``negative'' engine burns to mathematically cheat the $\Delta V$ calculation, which is physically impossible.
\end{itemize}

% ---------------------------------------------------------
\section*{4. The Callback Function (The ``Message Window'')}

\begin{lstlisting}[language=Python]
def print_iteration(x):
    # ... calculates current state ...
    print(f"Yukon Iteration... ")
\end{lstlisting}

When GMAT is running, it outputs iteration data into the Message Window. This Python function replicates that behavior. It allows us to peek inside the optimization loop every time the algorithm takes a step, letting us observe the \textbf{Active-Set method} in real-time. Initially, the solver often ignores the fuel cost entirely, taking large steps just to satisfy the altitude constraint. Once the constraint is met, it ``slides'' along the constraint boundary to find the cheapest fuel cost.

% ---------------------------------------------------------
\section*{5. Executing the Solver (The ``Optimize Loop'')}

This single function call is the exact equivalent of wrapping your GMAT sequence inside an \texttt{Optimize} and \texttt{EndOptimize} bracket.

\begin{lstlisting}[language=Python]
result = minimize(
    objective_function, 
    initial_guess, 
    method='SLSQP', 
    bounds=bnds, 
    constraints=cons, 
    options={'ftol': 1e-6}
)
\end{lstlisting}

\begin{itemize}
    \item \textbf{\texttt{method='SLSQP'}:} Sequential Least SQuares Programming. The Python equivalent of GMAT's Yukon.
    \item \textbf{\texttt{ftol} (Function Tolerance):} This parameter dictates when the algorithm is allowed to stop iterating. A value of $1 \times 10^{-6}$ translates to: \textit{``If you take a step, and the fuel cost changes by less than 0.000001, you have reached the absolute bottom of the valley. Stop running.''} If this number is set too high, the solver stops early and provides a highly inefficient trajectory. If it is too low, the solver will loop endlessly, chasing microscopic numerical rounding errors.
\end{itemize}
